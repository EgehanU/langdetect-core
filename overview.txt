# Language Detection Engine (langdetect-core)

## 1. Project Goal

Build a lightweight, performance-conscious language detection engine with a C++ core and a later Python wrapper.

The first version should be small, testable, and honest about confidence. The engine should detect:

* Script groups for non-Latin text
* A small set of Latin-script languages using n-gram profiles
* Unknown or ambiguous input when the text is too short or too close to call

The project avoids heavy ML models and large runtime dependencies. Performance matters, but correctness and measurable accuracy come first.

---

## 2. Scope (Version 1 Discipline)

### Included

* Unicode script detection
* Latin language detection using statistical n-grams
* Reusable C++ library
* CLI tool
* Profile generation pipeline

### Explicitly Excluded (for now)

* Mixed-language segmentation
* Dictionary-based scoring
* Multithreading
* SIMD optimization
* REST API
* WebAssembly
* Plugin/community system

These will only be added after correctness, evaluation, and baseline performance are proven.

## 4. Result Types

The engine must clearly distinguish between language detection and script detection.

### Language result

```json
{
  "kind": "language",
  "label": "de",
  "score": 0.82
}
```

### Script result

```json
{
  "kind": "script",
  "label": "cyrillic",
  "score": 0.97
}
```

### Unknown result

```json
{
  "kind": "unknown",
  "label": "unknown",
  "score": 1.0
}
```

### Ambiguous result

```json
{
  "kind": "ambiguous",
  "label": "ambiguous",
  "score": 0.51
}
```

This design avoids overclaiming confidence.

## 5. Detection Pipeline

### Step 1: UTF-8 Decoding

* Input is treated as UTF-8
* `std::string` is only a byte container -> must decode to Unicode code points
* Invalid UTF-8 must:

  * Not crash
  * Not dominate scoring

Handling:

* Replace invalid sequences or skip them
* If input is heavily corrupted -> return `unknown`

---

### Step 2: Script Detection

Script detection runs first because it is fast and often decisive.

Supported scripts:

* `latin`
* `cyrillic`
* `arabic`
* `greek`
* `devanagari`
* `thai`
* `korean`
* `japanese`
* `cjk_han`
* `unknown`

#### Classification rule

A script is selected if:

```text
> 70% of detected characters belong to that script
```

This is a tunable threshold. It should be adjusted later using evaluation data.

#### Special rules

* Presence of Hiragana or Katakana -> classify as `japanese`
* Hangul characters -> classify as `korean`
* Han-only text -> classify as `cjk_han`, not specifically Chinese

Important:

Do NOT classify Han-only text as Chinese. Japanese also uses Kanji, so this case is inherently ambiguous.

Japanese should be treated as a special override: if Hiragana or Katakana is present, the result can be `japanese` even when Han characters make up a large share of the text.

---

### Step 3: Latin Language Detection

Latin detection runs only if the text is mostly Latin.

#### Precondition

* If Latin characters < 70% -> skip Latin detection

This is also a tunable threshold. It should not be hardcoded forever without evaluation.

---

#### Normalization (Version 1)

Keep normalization simple and safe:

* Convert ASCII letters to lowercase
* Keep accented characters (do NOT strip them yet)
* Remove punctuation
* Collapse repeated whitespace
* Ignore emojis and symbols

Normalization of accents should be evaluated later, not assumed.

Important:

The runtime detector and the profile builder must use the same normalization rules. If runtime detection keeps accents but profile generation strips them, scoring quality will be damaged.

---

#### N-gram Extraction

* Use trigrams (initial implementation)
* Extract character-level trigrams from normalized text
* Count frequency

---

#### Scoring

* Compare input trigram frequencies with language profiles
* Compute similarity score per language
* Normalize scores so they sum to 1.0 (for ranking only)

These are **not true probabilities**.

---

### Step 4: Decision Logic

#### Minimum input threshold

```text
If normalized Latin letter count < 20 -> return unknown
If usable trigram count < 10 -> return unknown
```

---

#### Ambiguity rule

```text
If (top_score - second_score) < 0.05 -> return ambiguous
```

This is a tunable threshold. The value should be selected using evaluation data, not guessed permanently.

---

#### Additional rules

* If text is mostly digits/punctuation -> return unknown
* If script detection is strong -> return script result
* Only return language if confidence is sufficient

---

## 6. Handling Mixed Language Input

Version 1 does NOT implement full mixed-language segmentation.

Behavior:

* Analyze full input as a whole
* If signals conflict -> return `ambiguous`

Sliding-window detection can be added later.

---

## 7. Data Pipeline

Profile quality is the most important factor.

Pipeline:

1. Collect corpus text
2. Clean corpus
3. Normalize corpus
4. Generate n-gram frequencies
5. Save profiles
6. Evaluate on labeled test data

Important:

Do NOT rely on manually created profiles for real accuracy.

---

## 8. Training And Runtime Consistency

The profile builder and runtime detector must process text the same way.

The following logic should match between profile generation and runtime detection:

* UTF-8 decoding assumptions
* Latin normalization
* Accent handling
* Punctuation handling
* Whitespace collapsing
* N-gram size
* N-gram extraction rules
* Filtering of low-signal text

If training and detection use different preprocessing, the scores become unreliable.

## 11. C++ API Design

```cpp
enum class ResultKind {
    Language,
    Script,
    Unknown,
    Ambiguous
};

struct DetectionScore {
    std::string label;
    ResultKind kind;
    double score;
};

class LanguageDetector {
public:
    std::vector<DetectionScore> detect(std::string_view text) const;
};
```

Reasons:

* Preserves ranking
* Avoids unnecessary copying
* Explicit result type

## 16. Performance Strategy

Start single-threaded.

Focus on:

* Minimal copying
* `std::string_view` usage
* Cache-friendly loops
* Avoid unnecessary allocations
* Preloaded language profiles
* Compact profile representation
* Single-pass script counting
* Avoiding `std::map` in hot paths
* Avoiding repeated UTF-8 decoding of the same input

Do NOT add multithreading until benchmarks justify it.
Do NOT add SIMD until algorithm is stable.

Recommended version 1 performance design:

* Decode UTF-8 once per input.
* Count scripts in one pass over decoded code points.
* Run Latin normalization only if script detection says the text is mostly Latin.
* Store script counters in fixed-size structs or arrays, not hash maps.
* Store language profiles in memory after initialization.
* For the five initial languages, prefer vectors or fixed arrays over nested maps in the scoring hot path.
* Sort only the final small result list, not intermediate data.
* Keep JSON parsing/loading outside the detection hot path.
* Benchmark before optimizing beyond this.

Performance targets should be measured with repeatable benchmarks:

* Very short input: less than 100 characters
* Normal sentence: 100 to 500 characters
* Paragraph: 500 to 5,000 characters
* Large document: 100 KB and above

Version 1 does not need extreme optimization. It needs a design that does not block future optimization.

This project is a **modular, efficient, and extensible language detection engine** that combines:
